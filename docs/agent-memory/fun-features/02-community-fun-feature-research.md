### Leveling / XP from Message Activity (count-based)

- **Description:** Members earn XP for chat activity, counted per qualifying message (not by reading text), with a per-user cooldown. XP accumulates into levels via a curve; reaching configured thresholds can auto-grant 'level reward' roles. This is the anchor feature of the whole group — leaderboards, profile cards, achievements and voice XP all hang off the same XP store.
- **Discord mechanics:** Subscribe a module event handler to the existing `message.create` PlatformEvent (already surfaced in packages/core/src/contracts/events.ts). It fires WITHOUT any privileged intent, and we only need its metadata (author, guild, channelId, hasAttachments) — NOT `content` — so message-count XP works with the default Guilds intent. On each event: skip bots (author.bot), enforce a per-user 60s cooldown (last-award timestamp in a Postgres `user_xp` row, Drizzle), award a randomized 15-25 XP, recompute level from a stored curve, and on level-up optionally call the guild service `addRole` (same capability role-menus/birthdays already use) for reward roles and post/DM a configurable level-up message. `/rank` (slash) replies ephemerally or with a rendered card (see profile-card candidate). Admin page (Fastify SSR) configures cooldown, XP range, no-XP channel/role lists, level-up message, and reward-role thresholds via the module configSchema. Anti-farm: cooldown + per-channel/role denylist + ignore attachment-only/short-burst messages.
- **Example commands:** `/rank`, `/rank user:@member`, `/level-rewards list`, `(admin web) configure cooldown, XP range, no-XP channels, reward roles`
- **External deps:** None. Pure Postgres + existing message.create event + existing addRole guild-service capability.
- **Privileged intent:** No. Message-COUNT XP needs only the default Guilds intent because we never read message text; the MessageContent privileged intent is NOT required (and is OFF by default). Note: a future 'XP only for messages of length N / containing words' refinement WOULD need the privileged MessageContent intent.
- **Permissions:** ManageRoles (only if level-reward roles are enabled), SendMessages (for level-up announcements). Both already used by other modules.
- **Overlaps existing:** No existing leveling/XP module (confirmed: no level/xp/leaderboard code in packages/). Reuses addRole (role-menus/birthdays/welcome), the scheduler pattern, the repo/service/configSchema module shape, and message.create which the automod module already consumes. Conceptually adjacent to automod (both read message.create) but no functional overlap.
- **Recommendation:** **now** — This is the single highest-leverage engagement primitive for a large community and the foundation the rest of the group depends on, so it should land first. Leveling/XP is the flagship feature of every major community bot (MEE6, Arcane 2.3M+ servers, Tatsu) and is repeatedly described as a proven driver of sustained activity and friendly competition. The critical win for THIS bot: count-based XP needs NO privileged intent, so it is cheap and low-risk to ship. The real risk is farming, which is well-understood and mitigable with the cooldown + no-XP-channel + spaced reward-role pattern the research calls the proven default (60s cooldown reduced farming complaints 70%+). Scored funFactor 4 (a number alone is mild fun; the payoff is in rewards/cards/leaderboards), spamAbuseRisk 4 (farming is the known failure mode and must be designed against from day one).

### Server XP Leaderboards

- **Description:** Ranked board of members by XP/level, with all-time plus rolling weekly/monthly windows. Drives competition, recognition, and re-engagement ('I dropped to #6, better get active'). Includes a per-member rank lookup (your position out of N).
- **Discord mechanics:** Reads the same `user_xp` Postgres store as the leveling feature. `/leaderboard` (slash) returns a paginated embed; paging is handled with BUTTONS (Prev/Next/Jump) — well within the bot's button-only interactivity (no modals/reactions). A STRING SELECT MENU switches scope (All-time / This week / This month / Voice). For weekly/monthly windows, store per-period XP deltas (e.g. an `xp_period` table keyed by ISO week/month) OR snapshot totals; a Scheduler job (existing Scheduler in packages/core/src/scheduler.ts, same interval pattern as birthdays) rolls the period over and can post a 'Top 10 this week' recap to a configured channel. Optional: render the top-10 as a single PNG via the cards module for a richer post. Admin page sets the recap channel/cadence and which scopes are enabled.
- **Example commands:** `/leaderboard`, `/leaderboard scope:weekly`, `/leaderboard scope:voice`, `/rank (shows your position N of M)`, `(scheduler) weekly Top-10 recap post`
- **External deps:** None. Postgres queries + existing Scheduler + button/select pagination + optional cards render.
- **Privileged intent:** No (inherits from message-count XP; no text read needed).
- **Permissions:** SendMessages (for the scheduled recap post). No ManageRoles needed.
- **Overlaps existing:** None. Net-new, but tightly coupled to the leveling feature (cannot exist without it). Reuses Scheduler (birthdays/scheduled-messages pattern), button pagination, optional cards rendering.
- **Recommendation:** **now** — Leaderboards are nearly free once XP exists (it is mostly a query + button-paginated embed) and they are a primary reason leveling drives engagement — Arcane and MEE6 both center the /leaderboard and web leaderboard as the competitive/recognition payoff. Rolling weekly/monthly windows plus a scheduled recap are the proven mechanism for re-engaging lapsed members and giving newer members a winnable race (all-time boards calcify and discourage newcomers). Very low tech risk, high fit, high repeatability. Ship together with leveling as one coherent launch. spamAbuseRisk is low (read-only board) and the farming concern is handled upstream by the leveling cooldown/denylist design.

### Fun Profile Cards (/profile rendered image)

- **Description:** A personalized rank/profile card image: avatar, level, XP progress bar, rank position, and (if achievements ship) equipped badges. The visual, shareable identity layer that makes leveling feel rewarding rather than just a number.
- **Discord mechanics:** Reuse the EXISTING cards module's server-side rendering (packages/cards-module: buildCardSvg + renderCardPng via resvg — already producing PNGs for welcome/birthday cards). Add a profile layout that binds XP/level/rank/progress placeholders (the placeholder/layout system already exists). `/rank` or `/profile` (slash) fetches the user's XP row, renders the PNG, and replies with it as an attachment (AttachFiles permission, already declared by cards). No new external dependency — resvg, SVG layout, and the storage/placeholder pipeline are all in place. Customization (background/accent) can be exposed via STRING SELECT MENU choices or the admin page rather than free-text (no modals available). Avatar fetch is via the user's Discord avatar URL already present on PlatformUserRef.
- **Example commands:** `/profile`, `/rank (renders card instead of text)`, `/profile background:(select menu)`, `(admin web) preview/edit the profile card template`
- **External deps:** None new — reuses the in-repo resvg-based cards renderer. (Only caveat: avatar image fetch is an outbound HTTP to Discord's CDN, already an accepted pattern for welcome cards.)
- **Privileged intent:** No.
- **Permissions:** AttachFiles (already declared by the cards module).
- **Overlaps existing:** Reuses, does not duplicate, the dynamic-cards module (welcome/birthday cards). This adds a new card TEMPLATE + data binding, not a new renderer. Depends on the leveling feature for the data it displays.
- **Recommendation:** **later** — Highest fun-factor in the group — the rendered card is exactly what makes MEE6/Arcane/Tatsu leveling feel premium and is the most-shared artifact (people screenshot their cards), and THIS bot is unusually well-positioned because the resvg rendering pipeline already exists, making complexity/risk low. Marked 'later' rather than 'now' only because it is strictly a presentation layer on top of XP: ship leveling + leaderboards first (text/embed /rank is fine at launch), then add the card as a fast follow once the XP data is proven. It is a near-certain second-wave win, not a blocker. moderability 5 because the template is admin-controlled and content is the bot's own render (no user free-text to police); spamAbuseRisk 2 since /profile can be rate-limited per user.

### Achievements / Badges

- **Description:** Unlockable milestones (e.g. 'Reached level 10', 'Top 3 this week', '100 days active', 'Sent 1,000 messages') that grant a collectible badge shown on the profile card and optionally a Discord role. Adds goal-directed, completionist motivation on top of raw XP.
- **Discord mechanics:** An `achievements` definition table (admin-authored) + a `user_achievements` unlock table in Postgres. Evaluation is event/scheduler-driven: hook the same `message.create` award path to check count/level thresholds inline, and use a Scheduler job (existing pattern) for time-based ones (streaks, 'top N this week'). Unlock => DM/announce + optionally addRole (existing capability) + flag the badge for display on the profile card (rendered by the cards module). Browsing is button/select-driven: `/achievements` lists earned + locked with BUTTON pagination; a STRING SELECT MENU equips which badges show on the card. No modals/reactions needed. Admin page defines achievements (name, icon asset uploaded via existing card-asset storage, trigger type, threshold, optional reward role). MEE6's 'Almost There' nudge can be replicated as an ephemeral hint in /achievements.
- **Example commands:** `/achievements`, `/achievements user:@member`, `/badges equip (select menu)`, `(admin web) create achievement: trigger + threshold + icon + reward role`
- **External deps:** None. Postgres + existing scheduler + existing card-asset storage for badge icons.
- **Privileged intent:** No for activity/level/count/voice-derived achievements. Any achievement keyed on message TEXT (e.g. 'used a keyword', 'posted a link') would require the privileged MessageContent intent — avoid those in the initial set.
- **Permissions:** ManageRoles (if achievements grant reward roles), AttachFiles (badge icons on cards), SendMessages (unlock announcements).
- **Overlaps existing:** No existing achievement system. Leans on leveling (XP data), cards (badge display + asset storage), scheduler, and addRole. Highest internal-dependency surface of the group, which is why it is best built after leveling/leaderboards/cards exist.
- **Recommendation:** **later** — A strong, proven engagement layer (MEE6 ships custom badges + 'Almost There' nudges; the dedicated Achievements bot builds its whole value prop on badges + profile cards) that deepens retention by giving members concrete goals beyond a rising number. But it is the most complex item in the group: it needs an admin authoring UI, a definitions/unlocks data model, both inline and scheduled evaluation, and ideally the profile card already in place to display badges. That dependency chain (leveling -> cards -> achievements) plus techComplexity 4 / techRisk 3 (edge cases around retroactive unlocks, double-grants, period-based achievements) make it a clear 'later'. spamAbuseRisk 3 because activity-threshold achievements inherit XP-farming risk. Keep the launch set free of text-content triggers to avoid the privileged intent.

### Voice-activity XP

- **Description:** Award XP for time spent actively in voice channels (unmuted, not deafened, not alone/AFK), feeding the same level/leaderboard/profile system. Repeatedly cited as the single biggest engagement multiplier in real leveling deployments.
- **Discord mechanics:** PROBLEM for THIS bot: there are currently NO voice-state platform events surfaced to modules. packages/core/src/contracts/voice.ts exposes only VoiceCapability/VoiceSession for the audio PLAYER (join/play/pause), not member voice join/leave/mute-state events. So voice XP requires NEW adapter work: subscribe to discord.js voiceStateUpdate, define new PlatformEvent type(s) (e.g. voice.state.update), and surface them through the adapter to modules — a cross-cutting change beyond a single module. Once surfaced: track session start/stop per user, run a Scheduler tick (existing pattern) to credit XP per active minute, applying AFK/mute/deaf/solo-channel filters to prevent farming. Requires the GuildVoiceStates intent (NON-privileged, so cheaper than MessageContent/GuildMembers, but still must be enabled and plumbed). Writes to the same user_xp store; surfaces as a 'Voice' leaderboard scope and a profile-card stat.
- **Example commands:** `(passive) earn XP while active in voice`, `/leaderboard scope:voice`, `/rank (shows voice XP)`, `(admin web) voice XP rate, AFK/mute exclusions, eligible channels`
- **External deps:** None external, but requires internal ADAPTER work: new voiceStateUpdate -> PlatformEvent plumbing that does not exist today.
- **Privileged intent:** Not privileged, but DOES require enabling the GuildVoiceStates gateway intent AND new adapter event plumbing (voice-state events are not currently surfaced to modules). This is real extra cost even though the intent itself is non-privileged.
- **Permissions:** ViewChannel/Connect-level visibility for voice channels; ManageRoles only if voice milestones grant roles. SendMessages for any announcements.
- **Overlaps existing:** No overlap, but only partial reuse: the audio module's VoiceCapability is for playback, not member-state tracking, so the needed events must be built. Once built, it cleanly extends leveling/leaderboards/profile.
- **Recommendation:** **later** — Engagement upside is the highest in the group — research repeatedly calls voice XP the biggest single engagement multiplier and Arcane grew to millions of servers largely by offering free voice XP. So communityImpact/repeatability are 5. But for THIS codebase it is gated by missing infrastructure: voice-state events are not surfaced to modules, requiring new adapter plumbing and the GuildVoiceStates intent before any module logic can run (hence fitsExistingBot 3, techComplexity 4). The right sequence is: ship message-count leveling + leaderboards (no plumbing needed), then invest in the voice-state adapter work and add voice XP as a high-impact expansion. AFK/mute/deaf/solo farming filters are essential and add edge-case risk (spamAbuseRisk 4: AFK/idle farming is the classic voice-XP exploit). Clear 'later', high priority within 'later'.

### Reputation / Thanks (+rep, member-to-member kudos)

- **Description:** Lightweight member-to-member recognition: give another member a 'rep'/thanks point (with a per-day cooldown), shown on profiles and a separate rep leaderboard. Rewards being helpful/positive rather than just volume of messages — a social complement to activity XP. (ADDED candidate — surfaced by research; Tatsu's reputation system is a notable example.)
- **Discord mechanics:** Fully within the bot's existing capabilities. `/rep @member [reason]` (slash) with a per-giver daily cooldown enforced via a Postgres `reputation`/`rep_log` table; reason captured as a slash STRING OPTION (no modal needed — slash options sidestep the no-modal limitation). Self-rep blocked; bot targets blocked. Surfaces as a rep count on the profile card (reuse cards module) and a `/rep-leaderboard` with button pagination. Optional Scheduler job for a weekly 'most-appreciated member' recap. No message-text reading, so no privileged intent. Anti-abuse: daily cap per giver, optional minimum-account-age / minimum-level gate to prevent alt farming, and audit logging via the existing audit port.
- **Example commands:** `/rep user:@member reason:helped me debug`, `/rep-leaderboard`, `/profile (shows rep total)`, `(scheduler) weekly most-appreciated recap`
- **External deps:** None. Postgres + slash options + optional cards/scheduler.
- **Privileged intent:** No.
- **Permissions:** SendMessages (for any announcements). No ManageRoles required.
- **Overlaps existing:** No existing rep/kudos feature. Independent of leveling (can ship standalone) but composes nicely with profile cards and the leaderboard UI patterns. Distinct from XP because it measures peer recognition, not activity volume.
- **Recommendation:** **later** — A proven, low-cost identity feature (Tatsu's reputation/up-vote system is a recognized standout) that rewards quality and helpfulness — a useful counterweight to pure-volume XP and a natural fit for a large helpful community. It is technically easy and uses only slash options + buttons (no modals, no privileged intent, no reactions), so risk and complexity are low. Marked 'later' because it is additive and not part of the core leveling launch; it is a good early follow-on, especially since it can reuse the profile card and leaderboard scaffolding. spamAbuseRisk 3: alt/clique farming is the main concern, mitigated by daily caps and an optional level/account-age gate.

### Virtual currency / economy core (no real money)

- **Description:** A per-guild virtual point/currency system that is the backbone of every other Economy & Rewards feature. Members passively earn currency from chat activity (message.create counting) and active commands (/work, /daily), with balances, transfers (/pay), and a leaderboard. It is the 'faucet + ledger' layer; the shop, gambling, and pets are the 'sinks' that consume it. Currency is server-scoped and has no real-money or tradable-for-value component.
- **Discord mechanics:** Slash commands: /balance (ephemeral), /work (cooldown-gated faucet), /pay @user amount (transfer with confirm BUTTON), /leaderboard (paginated via Prev/Next BUTTONS), /economy admin subcommands for config. Passive earn: the bot already receives message.create events, so XP/currency-per-message works WITHOUT the privileged MessageContent intent because we only need the fact that a message occurred, not its text — apply per-user cooldown (e.g. 60s) in Postgres to throttle. Persistence: Drizzle tables economy_balances (guildId, userExternalId, balance, lifetimeEarned), economy_ledger (append-only for audit/anti-abuse), economy_settings (per-guild faucet rates, cooldowns, currency name/emoji, enabled). The existing scheduler (ScheduledJob intervalMs) can run periodic resets and a weekly leaderboard post. Admin web panel page for tuning rates and currency name fits the existing Fastify SSR per-module config pattern. Anti-abuse: per-user/per-channel cooldowns, bot-message exclusion, and a config cap on daily passive earnings. No modals or emoji reactions needed; buttons/selects only.
- **Example commands:** `/balance`, `/work`, `/daily`, `/pay @user 100`, `/leaderboard`, `/economy config rate:5 cooldown:60s name:Coins`
- **External deps:** None. Pure Postgres/Drizzle + scheduler. No external APIs.
- **Privileged intent:** No for the recommended design — passive chat earning uses message.create occurrence only, not message text, so the privileged MessageContent intent is NOT required.
- **Permissions:** None beyond default send/read. No role management in the core (that is the shop).
- **Overlaps existing:** No direct overlap. It is adjacent to a leveling/XP system but none exists yet; XP and currency can share the same message-count plumbing. Does not duplicate any of the 11 modules.
- **Recommendation:** **now** — This is the load-bearing prerequisite: the shop, gambling, pets, and streaks are all worthless without a currency ledger, and every major engagement bot (MEE6, Tatsu, Dank Memer, UnbelievaBoat) treats the economy core as the spine that keeps members returning daily. Fit is excellent — it needs only Postgres, the scheduler, slash commands and buttons, all of which exist, and crucially it avoids privileged intents by counting message occurrence rather than content. Fun on its own is modest (3) — a balance number is not exciting — which is why it scores high on community impact (5) and repeatability (5) rather than novelty. The real cost is abuse surface (4): alt-account farming and chat spam to mint currency are the well-documented failure modes, so per-user/per-channel cooldowns, earning caps, and an append-only ledger for moderator clawbacks are mandatory, not optional. Build this first and gate the rest of the group behind it.

### Daily / streak claim

- **Description:** A /daily command that grants a currency reward once per 24h, with an escalating streak bonus for consecutive days and a streak reset (or soft grace) on a miss. This is the single highest-ROI daily-engagement loop in the group and the canonical reason members open the server every day, mirroring Tatsu's t!daily and MEE6/Dank Memer dailies.
- **Discord mechanics:** Slash command /daily: checks economy_streaks (guildId, userExternalId, lastClaimDate, currentStreak, longestStreak) in Postgres, awards base + streak-scaled bonus, replies ephemerally or publicly with a streak counter and a 'come back in Xh' note. A milestone (e.g. 7-day) can grant a bonus shown with a celebratory embed. Optional /streak to view your record and a BUTTON-paginated streak leaderboard. The existing scheduler can run a daily tick to expire broken streaks deterministically (so logic is server-time-anchored, not dependent on when the user calls it). Depends on the currency core for the payout. No modals, no reactions — one slash command plus optional buttons. Anti-abuse is naturally low because the 24h cooldown is the rate limit; main risk is alt accounts, mitigated by the shared economy caps.
- **Example commands:** `/daily`, `/streak`, `/streak leaderboard`
- **External deps:** None. Postgres + scheduler only.
- **Privileged intent:** None.
- **Permissions:** None.
- **Overlaps existing:** No overlap. Conceptually similar to the existing reminders module only in that both are time-based, but functionally distinct. Does not duplicate any module.
- **Recommendation:** **now** — Daily/streak is the highest engagement-per-line-of-code feature in the entire group: research consistently shows daily login/claim mechanics with escalating streaks and a 'punishment' for missing drive measurable retention, and it is a flagship of Tatsu, MEE6 and Dank Memer. Technically it is trivial — one slash command, one small table, a deterministic daily scheduler tick that the codebase already supports, and it reuses the currency core. Abuse risk is genuinely low (2) because the 24h cooldown self-throttles. Its only weakness is that it is a thin loop without anything to spend on, so it must ship together with the currency core (and ideally the shop) to matter. I score fun 3 because the claim itself is a number, but communityImpact and repeatability are maxed because the habit it builds is exactly what a large server wants. Strong immediate candidate, paired with the core.

### Role / perk shop (buy roles with points)

- **Description:** A configurable shop where members spend earned currency on server rewards — primarily cosmetic/perk roles (colored names, custom flair, channel access) and optionally temporary perks. This is the essential currency SINK that gives the economy a point; without it, earned currency is meaningless and engagement decays.
- **Discord mechanics:** Admin configures shop items via slash command or the admin web panel: shop_items (guildId, label, roleId, price, stock, durationHours nullable, enabled). Members run /shop to see a paginated catalog rendered with STRING SELECT MENU (pick an item) + Buy BUTTON, or /buy item:<select>. On purchase the module debits the balance in a Postgres transaction and calls service.addRole(userExternalId, roleId, 'shop purchase') — the exact addRole/removeRole guild-service methods already used by the role-menus module are confirmed available. Temporary perks (e.g. 24h color role) are reclaimed by a scheduler job that calls removeRole when the purchase expires — a strong currency sink recommended by economy-bot guides to fight inflation. Requires the bot's Manage Roles permission and correct role hierarchy (same constraint role-menus already documents). No modals/reactions; select menus + buttons only. Audit each purchase via the existing audit port for moderator visibility and refunds.
- **Example commands:** `/shop`, `/buy color-purple`, `/shop admin add role:@VIP price:5000 stock:10`, `/shop admin add perk:custom-color duration:24h price:500`
- **External deps:** None.
- **Privileged intent:** None.
- **Permissions:** Manage Roles required, plus the bot's top role must sit above any purchasable role (role-hierarchy constraint, identical to the existing role-menus module). Admin should be warned in config UI, as a misconfigured hierarchy silently fails the addRole call.
- **Overlaps existing:** Partial conceptual overlap with role-menus (both assign roles via the same addRole plumbing) but the mechanic is fundamentally different: role-menus are FREE self-assign toggles; the shop is a PAID, debit-gated, stock-limited purchase with optional expiry. Not a duplicate — it is the monetized counterpart. Reuse role-menus' hierarchy-handling code.
- **Recommendation:** **now** — Every economy-bot guide reaches the same conclusion: without a meaningful sink, currency becomes worthless and members lose interest, so the shop is what makes the whole group cohere rather than an optional add-on. Fit here is unusually clean because the heavy lifting — addRole/removeRole with hierarchy handling — already exists and is proven in the role-menus module, so this is mostly a debit-then-assign transaction plus a paginated catalog using select menus and buttons the bot already supports. techRisk is a notch higher (3) than daily/streak only because of the role-hierarchy footgun and the need for atomic debit-and-grant (avoid double-spend on concurrent clicks), both solvable. Cosmetic colored-name roles and temporary perks double as inflation control. It should ship in the same wave as the core and daily; I mark it 'now' because the core without a sink is a half-built feature.

### Gambling-lite minigames (coinflip / slots) using points

- **Description:** Chance-based minigames — /coinflip (bet on heads/tails, ~2x payout) and /slots (spin a 3-reel emoji slot for tiered payouts) — wagering the server's virtual currency only. These are fast, repeatable dopamine loops that are a staple of Dank Memer, UnbelievaBoat and OwO, and they double as an aggressive currency sink. They are also the riskiest item in the group on safety and abuse grounds.
- **Discord mechanics:** Slash commands /coinflip side:<heads|tails> bet:<amount> and /slots bet:<amount>. Server-side RNG resolves the outcome, debits/credits the balance in a Postgres transaction, and replies with an animated-feel embed (slots reels as emoji); a 'Play again' BUTTON re-runs with the same bet. House edge configurable per guild (e.g. slots return-to-player < 100%) so it is a net sink, not a faucet. Strict per-user cooldown and a max-bet cap (economy_settings) are mandatory to prevent both spam and balance wipes. Depends on the currency core. No modals/reactions needed. Moderation: per-guild enable/disable (the platform already supports per-module enable), a max-bet cap, and ledger entries for every wager so mods can audit. IMPORTANT SAFETY NOTE: Discord's gambling policy only triggers on REAL-money / real-world-value wagers; purely virtual currency with no path to cash or tradable value is outside that policy — but simulated-gambling mechanics aimed at a large (potentially under-18) community are reputationally sensitive, so ship it OFF by default and clearly server-opt-in.
- **Example commands:** `/coinflip side:heads bet:100`, `/slots bet:250`, `/coinflip side:tails bet:50`
- **External deps:** None. Server-side RNG + Postgres.
- **Privileged intent:** None.
- **Permissions:** None.
- **Overlaps existing:** No overlap with existing modules.
- **Recommendation:** **later** — Gambling-lite is the most FUN item in the group (4) and the most repeatable (5) — coinflip/slots are proven engagement and currency-sink mechanics in Dank Memer and UnbelievaBoat — but it carries the highest spam/abuse risk in the whole group (5) and real reputational sensitivity, so it should not lead. The mechanics fit cleanly (server RNG, debit/credit transaction, buttons, per-guild disable all exist) and Discord's gambling policy specifically does NOT apply to non-cashable virtual currency, which de-risks the legal angle. But spam (rapid /slots farming), tilt (members wiping balances and getting upset, a documented community-management headache), and the optics of simulated gambling in a large, possibly underage server mean this needs the currency core, shop, robust cooldowns, max-bet caps, a configurable house edge, and OFF-by-default opt-in to exist responsibly first. Hence 'later': strong feature, but it must be built on top of a mature, well-moderated economy, not before it.

### Virtual pet / collectible (Tatsugotchi-style)

- **Description:** A long-term progression layer: each member adopts a virtual pet (or collects creatures) that they feed, play with, and level up over days using currency, unlocking cosmetics/collectibles. Modeled on Tatsu's Tatsugotchi and OwO's animal-catching, it adds deep, sticky retention beyond the daily loop — but it is by far the largest build in the group.
- **Discord mechanics:** Slash commands /pet adopt, /pet (view a server-rendered status card), /pet feed, /pet play, with hunger/happiness stats that decay over time via a scheduler tick and are restored by spending currency (another sink). Persistence: pets, pet_inventory, pet_collectibles tables. The status card is a natural fit for the EXISTING dynamic-cards module (resvg server-side PNG rendering) — a meaningful reuse win that makes the pet visually appealing without modals. Collectibles/gacha pulls resolve via server RNG and display with select-menu navigation + buttons. Interaction is entirely slash + buttons + selects; no modals or reactions. Depends on the currency core. The decay scheduler must be efficient (batch updates, lazy decay-on-read rather than per-pet ticks) to scale to a large server without hammering Postgres.
- **Example commands:** `/pet adopt name:Mochi`, `/pet`, `/pet feed`, `/pet play`, `/collection`, `/pet shop`
- **External deps:** None for logic. Strongly benefits from reusing the existing resvg dynamic-cards renderer for pet art; needs an art/asset set (sprites) which is a content cost, not a code dependency.
- **Privileged intent:** None.
- **Permissions:** None.
- **Overlaps existing:** No functional overlap, but it can REUSE the dynamic-cards (resvg) rendering pipeline for pet status cards — a fit bonus rather than a duplication concern.
- **Recommendation:** **later** — Virtual pets are the most FUN and most differentiating item in the group (fun 5): Tatsugotchi (100+ pets, housing, 2000+ item store) and OwO's collection grind are flagship retention features that hook members for weeks via long-term progression layered on the daily loop. The fit is partly favorable thanks to the existing resvg card renderer for pet art. BUT this is the heaviest build by a wide margin — techComplexity 5, techRisk 4 — requiring a stats/decay engine that must scale to a large server (use lazy decay-on-read, not per-pet scheduler writes), a sizeable art/content asset set, collectible/gacha balancing, and ongoing content updates to stay fresh. It is also the only item that demands meaningful design and art investment rather than just plumbing. Given that the entire economy group does not yet exist, pets are clearly a phase-2 'wow' feature to build AFTER the core/daily/shop loop is live and proven — hence 'later', not 'now'.

### Server lottery / raffle (scheduled jackpot draw)

- **Description:** ADDED CANDIDATE. A recurring server-wide lottery: members buy tickets with currency, the pot accumulates, and the scheduler draws a winner on a fixed cadence (e.g. daily/weekly). It is a powerful, low-effort communal currency sink and a recurring 'event' that pulls the whole server together at draw time — a proven faucet/sink balancer praised in economy-bot inflation guides and present in Dank Memer (lottery tickets).
- **Discord mechanics:** Slash command /lottery buy amount:<n> debits currency into a per-guild pot (lottery_tickets, lottery_pot tables). A /lottery view shows the current pot, odds, and your ticket count. The EXISTING scheduler (ScheduledJob intervalMs, already used by birthdays) runs the draw at the configured time, picks a weighted-random winner, credits the pot, posts an announcement embed to a configured channel (reusing the same sendMessage pattern the birthdays job uses), and records an audit event. Entirely server-side; no modals, no reactions. Depends on the currency core. Anti-abuse is inherently low: tickets cost currency (self-limiting), and alt-account risk is the same shared economy cap. Per-guild enable/disable and channel config fit the existing per-module admin panel pattern.
- **Example commands:** `/lottery buy 5`, `/lottery view`, `/lottery config channel:#general schedule:weekly`
- **External deps:** None.
- **Privileged intent:** None.
- **Permissions:** None beyond posting to the announcement channel.
- **Overlaps existing:** No overlap. Reuses the scheduled-draw + channel-announce pattern proven by the birthdays and scheduled-messages modules.
- **Recommendation:** **later** — I added this because research repeatedly flags communal sinks like lotteries as one of the cleanest ways to control currency inflation while creating a recurring server-wide event — and it maps almost perfectly onto infrastructure the bot already has: the birthdays/scheduled-messages modules already prove the 'scheduler tick -> pick targets -> post announcement embed -> audit' pattern this needs. It is low complexity (2), low risk, low abuse (tickets cost currency, which self-throttles), and delivers solid communal fun (4) and a strong sink. It is 'later' rather than 'now' only because it strictly depends on the currency core existing first and is a nice-to-have enhancement rather than a foundational piece; once the core/daily/shop wave ships, this is an easy, high-value follow-up that should beat both gambling and pets in build order on a cost/benefit basis.

### Trivia / Quiz (button answers, scheduled or on-demand)

- **Description:** Multiple-choice trivia where the bot posts a question as an embed with 2-4 lettered answer buttons. Members click their answer (acknowledged ephemerally so others don't see who picked what), the bot reveals the correct answer after a timer or after N responses, awards points, and tracks a per-guild leaderboard. Supports on-demand /trivia start and scheduler-driven 'trivia of the day/hour' drops into a configured channel. Categories and difficulty selectable via a string select menu.
- **Discord mechanics:** Maps almost perfectly onto existing primitives. Command /trivia (subcommands: play, leaderboard, scoreboard reset) via CommandDefinition. Question posted with GuildService.sendMessage / replyRich using OutgoingMessage.embed + buttons (one button per choice, customId like 'trivia:<roundId>:<choiceIdx>'). Answers handled in the component.interaction event handler exactly like role-menus' handleInteraction (parseCustomId pattern). event.reply() gives each player a private 'Locked in!' ack; first-correct or timer resolves the round and the bot edits the original message via GuildService.editMessage to reveal results. Scheduled drops use the existing Scheduler (interval job per guild, or runNow). Scores/rounds/leaderboard stored in a new Drizzle repo (guild_id, user_id, points, streak, answered_at) — trivially cheap. Category/difficulty pickers use a string select menu. Admin page can expose channel, schedule cadence, points-per-correct, and time limit via configSchema. No modals, no reactions needed.
- **Example commands:** `/trivia play category:Science difficulty:medium`, `/trivia leaderboard`, `/trivia daily channel:#general time:18:00`, `/trivia scoreboard-reset`
- **External deps:** Optional but recommended: Open Trivia DB (opentdb.com) — free, no API key, CC-BY-SA, 3000+ questions, supports session tokens to avoid repeats; rate limited to 1 req / 5s per IP (cache a batch per guild to stay well under). Can also ship a bundled/admin-authored question bank in Postgres for zero external dependency, which removes the only fragility.
- **Privileged intent:** None. Answers are button clicks, not chat messages — no MessageContent intent required. This is a major advantage over chat-typed quiz designs.
- **Permissions:** Standard SendMessages / EmbedLinks (Send Messages + Embed Links) in the trivia channel; nothing privileged.
- **Overlaps existing:** No direct overlap. No existing trivia/quiz/XP module. Complements (does not duplicate) custom-commands and scheduled-messages. Big bots bundle trivia alongside leveling, but this bot has no leveling system, so the trivia leaderboard is net-new value rather than a duplicate.
- **Recommendation:** **now** — Highest fit-to-fun ratio in the group. Trivia is a staple engagement driver in the bots this project benchmarks against (MEE6, in 21M+ servers, bundles a trivia plugin; trivia is repeatedly cited as an easy way to spark friendly competition and conversation with team play and weekly leaderboards). It needs nothing the platform lacks: buttons, select menus, scheduler, and Postgres are all already first-class, and it explicitly avoids the privileged MessageContent intent by using button answers. Anti-abuse is easy (one answer per user per round enforced in the handler, per-channel cooldown via scheduler/last-drop timestamp). Pure-button design plus pure-logic scoring makes it highly unit-testable. Recommend building now, with a bundled question bank as the default source and OpenTDB as an optional enrichment to avoid external fragility.

### Counting game (collaborative counting channel)

- **Description:** A dedicated channel where members count upward one number per message, no user counting twice in a row; the bot validates each number, reacts/acknowledges correct counts, resets to 1 (or punishes) on a mistake, tracks the all-time high and per-user contributions, and can grant milestone roles at 100/1000/etc.
- **Discord mechanics:** Fundamentally requires reading the TEXT of each message to know which number was typed — that is the whole game. The bot already receives message.create events, but MessageCreateEvent.content is an empty string unless the privileged MessageContent intent is enabled (it is OFF by default). So this feature is BLOCKED on enabling MessageContent, which is a privileged intent requiring Discord verification at scale and is called out as an explicit cost/risk in this project. There is no button/select redesign that preserves the feel of counting (a 'click +1' button degrades it into a clicker and loses the typing-the-number social ritual). If built: message.create handler parses content, compares to expected next number and last-counter user id, posts/edits feedback via GuildService, persists current count + last user + high score in a Drizzle repo, and uses GuildService.addRole for milestone roles. Note the bot has no emoji-reaction capability, so the genre-standard '✅ on each correct count' is not available — feedback would have to be a message edit or react-less, which is slightly worse UX.
- **Example commands:** `/counting setup channel:#counting`, `/counting stats`, `/counting leaderboard`, `/counting reset`
- **External deps:** None.
- **Privileged intent:** REQUIRED: privileged MessageContent intent (OFF by default) — the game cannot read typed numbers without it. This is the dominant cost and the reason this is not a 'now'. Also wants GuildMessageReactions (no reaction listener exists today) for the conventional ✅ feedback; degraded UX without it.
- **Permissions:** SendMessages; ManageRoles only if milestone roles are used; ideally ManageMessages to delete invalid counts. Plus the privileged-intent gate above.
- **Overlaps existing:** No existing counting module. No overlap with current features, but also no shared infrastructure to lean on for message-content reading.
- **Recommendation:** **later** — Genuinely sticky and self-sustaining where it lands — dedicated counting communities reach 1M+ counts and use milestone roles and multiple counting variants — so repeatability is excellent and it's a real engagement format. But it is the single worst architectural fit in the group for THIS bot: it is hard-blocked on the privileged MessageContent intent (and ideally a reactions intent the platform doesn't have), both flagged as extra-cost items. For a large community, enabling MessageContent has verification and privacy implications that should be a deliberate decision, not a side effect of a minigame. Defer until/unless the server already enables MessageContent for another reason (e.g. word-trigger automod); at that point the incremental cost drops to near zero and it becomes attractive. Marked 'later', not 'reject', because the demand is real and the only blocker is an intent toggle.

### Button reaction-race / fastest-click games

- **Description:** The bot posts a message with a button ('First to click wins!') after a random delay or countdown; the first member to click is declared winner and scored. Variants: 'don't click the bomb', whack-a-mole grids of buttons, or a draw-style fastest-finger round tied to giveaways.
- **Discord mechanics:** Technically simple with existing primitives: GuildService.sendMessage with a single button (customId 'race:<roundId>'), first component.interaction event for that customId wins, recorded in a Drizzle repo, message edited to announce the winner. Scheduler can launch surprise rounds. BUT there is an inherent fairness problem on Discord: button clicks are gateway interactions with variable network/processing latency (commonly 100s of ms), so 'fastest click' actually measures latency + luck, not skill — community tooling notes pure click-speed modules are 'not that accurate' because of interaction round-trip time. Also, only the platform-received order is observable, which can feel arbitrary to players. No modals/reactions needed, so the integration cost itself is low; the problem is the format, not the plumbing.
- **Example commands:** `/race start`, `/race bomb`, `/race leaderboard`
- **External deps:** None.
- **Privileged intent:** None (button clicks only).
- **Permissions:** Standard SendMessages.
- **Overlaps existing:** Mechanically near-identical to a button-entry giveaway draw (the scheduler+button-claim pattern the project already anticipates for giveaways). If a giveaway feature is ever built, 'fastest click' is largely a reskin and would feel redundant.
- **Recommendation:** **reject** — Weakest candidate in the group. The fun ceiling is low and the format is known to be unfair on Discord: button interaction latency means it rewards connection speed, not reflexes (acknowledged even by purpose-built click-speed modules), and the broader giveaway ecosystem has explicitly moved away from pure 'first to click' toward task/scarcity/referral mechanics because raw click-races don't drive durable engagement. Abuse risk is comparatively high (autoclicker scripts, multi-account sniping) with no clean mitigation. It adds little over a button-claim giveaway the platform may build anyway. Low effort to build, but low payoff and poor fairness — reject as a standalone feature; if anything, fold a fair, randomized button-claim into a future giveaway feature instead.

### Word / guess games (Hangman + Wordle-style)

- **Description:** Hangman (the bot picks a word, shows blanks, players guess letters; wrong guesses advance the gallows) and Wordle-style 5-letter guessing with green/yellow/grey feedback. Single-player solo runs and shared 'everyone guesses together' rounds, with optional streaks and a leaderboard.
- **Discord mechanics:** Letter input is the crux. The clean, intent-free design is an on-screen keyboard: render A-Z (and Enter/Backspace) as button rows; each click is a component.interaction with customId 'wordle:<gameId>:<letter>', the handler updates game state in Postgres and edits the message via GuildService.editMessage to redraw the board (green/yellow/grey rendered with emoji squares or via the existing resvg card renderer for a richer grid image). Hangman works the same with a letter grid plus an embed/image of the gallows. This avoids the privileged intent entirely. The chat-typed alternative ('/guess' would need a value — fine via a slash option, but free typing of guesses in-channel would need MessageContent). Discord's button cap (5 buttons x 5 rows = 25 components) comfortably fits a 26-letter keyboard if Enter/Backspace share a row or letters are split across two messages/select menus; a string select menu of letters is an ugly fallback. Scheduler enables a 'daily word' with shared streaks. resvg (cards-module) is a real asset for a polished Wordle grid image.
- **Example commands:** `/wordle daily`, `/wordle play`, `/hangman start difficulty:hard`, `/wordle leaderboard`, `/wordle streak`
- **External deps:** None required — ship a curated word list in Postgres/asset file (and for Wordle a separate answer list). Avoid scraping NYT. No live API needed.
- **Privileged intent:** None IF designed with a button keyboard or slash-option guesses (recommended). Only the 'free-type your guess in chat' variant would need the privileged MessageContent intent — avoid that variant.
- **Permissions:** Standard SendMessages / EmbedLinks; AttachFiles if using the resvg grid image.
- **Overlaps existing:** No existing word-game module. Note an external competitive reality: Wordle is now an official, NYT-licensed Discord Activity launchable with /wordle, so a custom clone competes with the first-party experience — differentiate via shared-server streaks/leaderboards and arbitrary word lengths/themes rather than copying daily NYT puzzles. Reuses the cards-module resvg renderer (nice synergy, not overlap).
- **Recommendation:** **later** — Solid fun and strongly buildable on this stack — pure game logic (great testability), no privileged intent if you use a button keyboard, and a natural reuse of the existing resvg card renderer for an attractive grid. The reasons it's 'later' not 'now': the button-keyboard UI is fiddly (component-cap juggling for 26 letters, multi-message or paginated input) so it's the most UI-intensive build in the group, and the daily-Wordle niche is partly occupied by Discord's official NYT Wordle Activity. Hangman has fewer such concerns and could ship first. Strong second-wave feature once trivia (which shares the leaderboard/round/scheduler scaffolding) is in place — build trivia first, then reuse that infrastructure here.

### Tabletop minigames (Rock-Paper-Scissors, Tic-Tac-Toe, Blackjack, Connect Four)

- **Description:** Classic quick games rendered entirely as buttons: RPS (1v1 or vs-bot, both pick via buttons), Tic-Tac-Toe and Connect Four (a button grid where players alternate moves), and Blackjack vs the dealer (Hit/Stand/Double buttons). Optional win tracking and a casual leaderboard.
- **Discord mechanics:** The textbook use case for this bot's component model. A Tic-Tac-Toe board is a 3x3 grid of buttons (well under Discord's 25-component cap); each click is a component.interaction with customId 'ttt:<gameId>:<cell>', the handler validates turn ownership (compare event.user.externalId to the stored current-player), updates state in Postgres (or even in-memory for ephemeral matches), and edits the message via GuildService.editMessage/event.update to redraw the board and switch turns. RPS uses three buttons and reveals on both picks. Blackjack uses Hit/Stand/Double buttons with the embed showing the hand. Challenge flow: /ttt @user posts a 'X invites Y' message with Accept/Decline buttons. No modals, no reactions, no privileged intents — only button clicks and message edits, all of which the platform already does in role-menus and audio control panels. Win/loss records persist in a small Drizzle repo. Scheduler not needed (on-demand).
- **Example commands:** `/ttt @user`, `/rps @user`, `/connect4 @user`, `/blackjack bet:100`, `/games stats`
- **External deps:** None — pure self-contained game logic.
- **Privileged intent:** None. Entirely button-driven; no MessageContent, no GuildMembers, no reactions.
- **Permissions:** Standard SendMessages / EmbedLinks only.
- **Overlaps existing:** No existing minigame module; no overlap with current features. These are ubiquitous in the fun bots this project benchmarks (Proto, Play Pal, Mini all ship TTT/RPS/Connect4/Blackjack), so they're table-stakes 'fun' rather than novel — but absence from this bot makes them clearly additive.
- **Recommendation:** **now** — Best pure technical fit in the group and very low risk: 100% button-and-edit, zero external dependencies, zero privileged intents, and the game logic is deterministic and trivially unit-testable (techRisk 1). RPS/TTT can be in-memory; only opt-in win tracking needs Postgres. These classics are a near-universal offering in popular fun bots (Proto, Play Pal, Mini), so members expect them and they reliably generate 1v1 banter. Start with the cheapest crowd-pleasers (RPS + Tic-Tac-Toe) to validate the challenge/turn-engine, then add Connect Four (same grid engine) and Blackjack. Flag: a betting/currency-based Blackjack would imply an economy system the bot doesn't have — ship Blackjack as point-less or with a tiny self-contained casual-points table to avoid scope creep. Recommend building now alongside trivia; it shares no infra with trivia so the two can proceed in parallel.

### Daily Word Puzzle with streaks (added candidate, Wordle-style streak engine)

- **Description:** An explicitly daily, streak-driven puzzle: one word per day per server (or per user), button-keyboard input, public 'who solved it / in how many guesses' results, and a persistent solve-streak + leaderboard. Distinct from generic word games by being a scheduled daily ritual (like the daily-Wordle modes that drive return visits) rather than on-demand.
- **Discord mechanics:** Same button-keyboard + GuildService.editMessage rendering as the Wordle candidate above, but the differentiator is the Scheduler: a daily job posts the puzzle to a configured channel and a daily reset rolls the answer and decrements streaks for non-solvers. Streaks, daily results, and leaderboards live in a Drizzle repo keyed by guild_id + date. Per-user solve attempts are tracked so each member gets one daily game (the documented 'one game per day, biggest streak' mode). Uses configSchema for channel + post time. No privileged intent (button keyboard); resvg renders the shareable grid image.
- **Example commands:** `/dailyword setup channel:#daily-puzzle time:09:00`, `/dailyword play`, `/dailyword streak`, `/dailyword leaderboard`
- **External deps:** None — bundled curated answer/word lists; do not reuse NYT puzzles.
- **Privileged intent:** None with the button-keyboard design.
- **Permissions:** SendMessages / EmbedLinks; AttachFiles for the grid image.
- **Overlaps existing:** Overlaps the Word/guess-games candidate above — this is essentially its 'daily + streaks' productization, and would be built as a mode of that module rather than a separate one. Competes with Discord's official NYT Wordle Activity, so lean on cross-member streaks/leaderboards (which the Activity doesn't surface per-server) as the differentiator.
- **Recommendation:** **later** — Added because the listed 'word/guess games' lumps on-demand play with the genuinely different, retention-focused daily-streak format, and daily streaks are a proven return-driver (documented 'one game per day, biggest streak' Wordle modes keep chats active day over day). Excellent repeatability and almost no abuse surface (one game per user per day). It's 'later' rather than 'now' because it depends on first building the word-game rendering/engine and because the daily-puzzle niche overlaps Discord's first-party Wordle Activity. ### Conversation Starters / Would-You-Rather / Question of the Day

- **Description:** Scheduled and on-demand prompts that lower the barrier to talking: a daily auto-posted Question of the Day plus on-demand /wyr (would-you-rather), /qotd, and never-have-I-ever / most-likely-to style prompts drawn from curated decks. The single most recommended tactic for keeping quiet servers active because it gives silent members an easy entry point.
- **Discord mechanics:** Outstanding fit. The existing interval scheduler (used by birthdays) posts the daily QOTD to a configured channel at a configured hour, optionally pinging a QOTD role and creating a thread for answers. On-demand /wyr and /qotd pull a random row from a Postgres questions table (seeded with a built-in pack; admins add custom ones). Would-You-Rather renders the two options as two BUTTONS and tallies votes live, editing the embed via component.interaction.update to show a vote split (the exact role-menus button pattern). Per-user vote dedup in Postgres. Admin page configures channel, schedule hour, active decks, and role to ping, all using existing ConfigFieldSchema field types (channel/role/select/number). Member-submitted questions go through /qotd suggest <string> into a review queue mods approve with Approve/Reject BUTTONS (mirrors the confession review flow). NOTE: a single-line slash string option is fine for question text; no modal needed. No privileged intent, no reactions.
- **Example commands:** `/qotd`, `/wyr`, `/qotd suggest question:What is the most overrated movie?`, `/qotd config channel:#daily-questions hour:9 role:@QOTD`, `/nhie`
- **External deps:** None. Questions ship as a seeded local pack (like Would You / QOTD Bot decks). No fragile external API.
- **Privileged intent:** No.
- **Permissions:** SendMessages, plus Create Public Threads if the auto-thread option is enabled.
- **Overlaps existing:** Slight surface overlap with scheduled-messages (cron posts) and announcements, but those post FIXED admin text; this rotates randomized prompts from decks with voting and member submissions, which scheduled-messages cannot do. Build as its own module rather than stretching scheduled-messages.
- **Recommendation:** **now** — Research is unambiguous that daily questions / WYR are a top-tier engagement driver in large servers (Would You, QOTD Bot, Truth or Dare Bot are built around exactly this) and they lower the participation barrier for lurkers. It rides entirely on infrastructure that already exists (scheduler, buttons, Postgres, admin config) with no intent cost and low abuse risk. The single strongest candidate in this group.

### Confessions (anonymous, moderated)

- **Description:** Members submit anonymous confessions via /confess; the bot posts them (stripped of author identity) to a confessions channel, optionally after moderator review. A staple of large social servers, with per-user anon-IDs, word filtering, a report button, and a private mod log mapping confessions back to authors for accountability.
- **Discord mechanics:** Mostly a strong fit with one UX caveat. /confess text:<string> takes the confession as a slash string option; the invocation and the bot ephemeral submitted reply are private, so anonymity holds. The bot posts the confession as an embed via GuildService.sendMessage with allowMentions fully disabled (the contract already defaults to no @everyone/role pings, important for anon safety). Review mode queues the confession in Postgres and posts it to a mod channel with Approve / Reject / Ban-author BUTTONS handled through component.interaction (same as the QOTD/quote approval flow); on approve it sends to the public channel. A Report BUTTON under each public confession lets members flag it. Per-user rotating anon-ID and a private author map in Postgres satisfy accountability without breaking anonymity. Automod banned-word logic can be reused to auto-flag confessions. CAVEAT: with no modal support, long confessions are entered in a single-line slash text field, more awkward than the multi-line modal ConfessionBot/Confessions use, usable but a noticeably worse authoring UX. No privileged intent and no reactions needed.
- **Example commands:** `/confess text:I have never actually watched Star Wars`, `/confess config channel:#confessions review:on`, `/confess ban`, `Report button under each posted confession`
- **External deps:** None.
- **Privileged intent:** No.
- **Permissions:** SendMessages in the confessions and mod-log channels; Manage Messages if mods delete posted confessions via the bot.
- **Overlaps existing:** No existing overlap. Reuses automod word-filter logic and the same Approve/Reject button-queue pattern as QOTD suggestions and quotes, shared infrastructure rather than duplication.
- **Recommendation:** **later** — Genuinely popular and engaging (Confessions is a top-tier social bot) and technically buildable on existing primitives. But it carries the highest abuse/safety surface in this group for a LARGE server (harassment, doxxing, NSFW, and the moderation burden of anonymous content), so it MUST ship with review mode, word filtering, rate limits, a report button, and a private author audit log. The missing modal also degrades the core authoring UX. Worth doing, but after lower-risk social features land and the button-queue + automod reuse is proven. Hence later, not now.

### Ship / Marriage social fun

- **Description:** Playful relationship mechanics: /ship @a @b returns a randomized compatibility percentage with a ship name and a love-meter bar; /marry @user proposes (accept/decline via buttons) and records a non-romantic, opt-in marriage shown on profiles, with /divorce to end it. Light, meme-y social entertainment popularized by Mudae and RelationshipBot.
- **Discord mechanics:** Clean fit. /ship @a @b is stateless fun: a percentage deterministic from the pair of IDs (so it is stable and not re-rollable), a generated ship name, and a love-meter rendered with emoji/text or a card via the existing resvg dynamic-cards renderer. /marry @user posts a proposal embed with Accept / Decline BUTTONS routed through component.interaction.reply, and on accept writes a marriage row to Postgres; /marriages and /divorce read/clear it. A per-guild marriage limit and a cooldown on /ship spam live in the service layer. Mention-only targeting via the user option type. No privileged intent, no reactions, no modals. Could optionally feed a married field into profile/card output later.
- **Example commands:** `/ship @alice @bob`, `/marry @user`, `/divorce`, `/marriages @user`, `/ship me @user`
- **External deps:** None.
- **Privileged intent:** No.
- **Permissions:** SendMessages. None special.
- **Overlaps existing:** No overlap. Optional future tie-in to dynamic-cards (love-meter image) and a profile surface, but stands alone.
- **Recommendation:** **later** — Fun and trivially cheap to build, and the /ship percentage is a classic icebreaker. But its community value is shallow: it is novelty entertainment, not a retention loop, and it can become low-grade spam or unwanted social pressure (shipping or forced marriage of real members can be awkward or harassing if not opt-in for marriages and rate-limited for ship). Pair-deterministic results and cooldowns mitigate spam. Good filler to bundle into a future fun/social pack, but not a priority build, hence later.

### Quotes & inside-jokes database

- **Description:** A server quote book: members save memorable lines with /quote add (text + attributed author), then recall them with /quote random, /quote view <id>, or /quote search, and browse a leaderboard of most-quoted members. Preserves a community's inside jokes and culture, a known retention asset because shared in-jokes deepen belonging.
- **Discord mechanics:** Good fit with one ergonomic caveat. /quote add text:<string> author:@user stores a quote in Postgres; /quote random and /quote view <id> return it as an embed; /quote search query:<string> does a SQL ILIKE/full-text lookup; /quote leaderboard ranks most-quoted members; paginated browsing via Prev/Next BUTTONS (role-menus pattern). Mods /quote delete <id>. CAVEAT 1 (UX): mainstream quote bots (Quoter, QuoteBot) capture quotes by RIGHT-CLICK > Apps context menu on an existing message, which this bot does not support (no application command context menus / message commands surfaced in the adapter), so quotes here are entered by PASTING text into a slash option rather than one-click-saving an existing message, functional but a worse capture UX. CAVEAT 2: auto-capturing a referenced message's text would need the privileged MessageContent intent (off), avoided entirely by the paste-the-text approach. No reactions needed. The single-line slash field also limits very long multi-line quotes (no modal), a minor authoring limitation.
- **Example commands:** `/quote add text:it is not a bug it is a feature author:@dave`, `/quote random`, `/quote view 42`, `/quote search query:bug`, `/quote leaderboard`
- **External deps:** None.
- **Privileged intent:** No for the paste-text design. Auto-capturing an existing message's content WOULD require MessageContent (privileged, off), explicitly avoided.
- **Permissions:** SendMessages. None special.
- **Overlaps existing:** No overlap with custom-commands (those are admin-authored canned responses; quotes are member-contributed, randomized, searchable, attributed). Distinct module.
- **Recommendation:** **later** — Beloved in established communities and very cheap on this stack (slash + Postgres + buttons) with low abuse risk. It is later not now because its value compounds only in mature servers with existing culture, and without context-menu capture or modals the save UX is meaningfully clunkier than the bots people know (Quoter/QuoteBot), dampening adoption. A solid second-wave social feature, ideally after the bot gains application-command context menus to make capture one-click.

### Starboard (highlight popular messages)

- **Description:** Community-driven highlight reel: when a message reaches a threshold of star reactions, the bot mirrors it (content, author, jump link, star count) into a #starboard channel, democratic pins with no limit. One of the most universally loved engagement features in large servers.
- **Discord mechanics:** POOR fit as classically expected; this is the misfit of the group. Every mainstream starboard (Carl-bot, Dyno, Starboard, Cakey) is built on the EMOJI-REACTION model: users react with a star, the bot counts reactions, mirrors at a threshold, and live-updates the count. This bot has NO reaction listener, NO GuildMessageReactions intent, and no reaction event in PlatformEvent, so the defining UX is impossible without new adapter + intent + event-type work (significant). A degraded button-based substitute (an upvote BUTTON under every message) is unworkable because the bot does not post most messages and cannot retroactively add components to arbitrary user messages. SECOND blocker: mirroring the original message's TEXT into the starboard requires reading message content, i.e. the privileged MessageContent intent (off). editMessage exists for live count updates, so the OUTPUT side is fine, but the INPUT side (detect reaction threshold + read content) needs two pieces this platform deliberately lacks. A faithful starboard therefore needs (1) GuildMessageReactions intent + a new reaction PlatformEvent + adapter wiring, AND (2) the privileged MessageContent intent.
- **Example commands:** `/starboard config channel:#starboard threshold:5 emoji:star`, `/starboard top`
- **External deps:** None external, but requires internal platform work: a new gateway intent (GuildMessageReactions), a new adapter-neutral reaction event, AND the privileged MessageContent intent.
- **Privileged intent:** YES, TWO costs: (a) GuildMessageReactions intent to count star reactions (and there is currently no reaction listener/event at all, so adapter + PlatformEvent work too), and (b) the privileged MessageContent intent (off by default) to mirror the starred message's text. The most intent-expensive feature in the group.
- **Permissions:** SendMessages and Manage Messages in the starboard channel; Read Message History to fetch starred messages.
- **Overlaps existing:** No feature overlap, but it would be the first consumer of a reaction-event pipeline that does not exist yet, so building it pulls in cross-cutting adapter work that would benefit future reaction features too.
- **Recommendation:** **later** — On pure engagement merit this is a 5/5: starboards are among the most loved features in large communities and drive a virtuous be-funny / get-starred / get-seen loop. But it is the worst architectural fit here: the platform deliberately has no reaction intent/listener and keeps MessageContent off, and a faithful starboard needs BOTH. It cannot be approximated well with buttons because the bot cannot attach components to arbitrary user messages. Recommend later, explicitly gated on a decision to add the GuildMessageReactions reaction pipeline (which would also unlock other reaction features) and to enable MessageContent. Not reject because the payoff justifies that investment eventually; not now because the cost is real and front-loaded.

### Party Games pack (Truth or Dare / Never Have I Ever / Most Likely To)

- **Description:** ADDED CANDIDATE. On-demand interactive icebreaker games: /tod (truth or dare), /nhie (never have I ever), /mlto (most likely to), each posting a prompt from a curated deck with BUTTONS to draw the next one or switch mode. Research surfaced Truth or Dare Bot as a top engagement bot built entirely on this loop; it is the natural companion to QOTD/WYR and a proven activity driver.
- **Discord mechanics:** Excellent fit and almost free if built alongside the QOTD/WYR module (shared deck table + button-draw engine). /tod posts a prompt embed with Truth / Dare / Next BUTTONS; component.interaction.update edits the message to the next prompt, so one message becomes an endless game without channel spam. /nhie and /mlto reuse the same engine with different decks; most-likely-to renders Vote BUTTONS tallied in Postgres. Decks ship seeded locally (no external API) and admins add custom prompts via /tod suggest, reusing the same approve/reject queue as QOTD. No privileged intent, no reactions, no modals.
- **Example commands:** `/tod`, `/nhie`, `/mlto`, `/tod config decks:sfw`, `Next / Truth / Dare buttons under the prompt`
- **External deps:** None, seeded local decks.
- **Privileged intent:** No.
- **Permissions:** SendMessages.
- **Overlaps existing:** Heavily shares infrastructure with the QOTD/WYR candidate (deck table, random-draw, button engine, suggestion queue), so strongly recommend building them as ONE engagement-prompts module with multiple game modes rather than two modules. Needs an SFW/NSFW deck toggle and moderation since prompt content can be edgy.
- **Recommendation:** **now** — Top-tier fun-factor with a self-contained, endless button-driven loop that maps perfectly onto this bot's component model and shares almost all its code with the QOTD/WYR candidate, making marginal cost near zero. Proven by Truth or Dare Bot's popularity. The main caveat is content moderation (edgy prompts), handled with an SFW-default deck toggle. Recommend shipping it in the same module as QOTD/WYR for maximum engagement per unit of work.

### Giveaways (button entry + scheduled draw)

- **Description:** Host giveaways where members enter by clicking a single button on an embed; the bot tracks entrants in Postgres and, at a scheduled end time, randomly draws one or more winners and announces them. Supports requirement gating (must hold a role), multiple winners, reroll, and an admin dashboard for active/past giveaways.
- **Discord mechanics:** /giveaway start (prize, winners, duration, optional required-role) posts an embed via GuildService.sendMessage with one 'Enter' button (customId 'giveaway:<id>'). Each click arrives as a ComponentInteractionEvent; the handler uses event.userRoleIds for requirement gating (no extra fetch) and records the entrant in a giveaway_entries table (unique on giveaway_id+user, so re-clicks are idempotent and double-entry is impossible). A scheduler job on the existing 30s delivery-tick pattern (see announcements DELIVERY_TICK_MS) polls for giveaways past end_at, draws winners with crypto RNG, and edits the original message + posts a winner announcement. event.reply() gives an ephemeral 'You're entered!' confirmation. /giveaway reroll and /giveaway end early are admin subcommands. Live entrant count is updated by editMessage on a throttle. No modal, no privileged intent, no emoji reaction needed — button entry is exactly the modern GiveawayBot model.
- **Example commands:** `/giveaway start prize:"Nitro Classic" winners:1 duration:24h required_role:@Member`, `/giveaway end giveaway_id:42`, `/giveaway reroll giveaway_id:42`, `/giveaway list`
- **External deps:** None. Pure Postgres + scheduler + buttons. No external API.
- **Privileged intent:** No. Button clicks arrive as component interactions; entrant identity comes from the interaction itself, not from message-content or member-join events.
- **Permissions:** SendMessages, EmbedLinks. If using required-role gating it only reads userRoleIds (already on the event). No role-granting needed unless a 'winner gets role' option is added (then ManageRoles + hierarchy check via canManageRole).
- **Overlaps existing:** No real overlap. Announcements/scheduled-messages handle one-way broadcasts, not interactive entry+draw. Reuses the announcements scheduler-job pattern and the role-menus component-interaction pattern almost verbatim.
- **Recommendation:** **now** — Highest-ROI candidate in the group. Giveaways are the single most cited engagement/growth driver for large servers (surveys cite ~23% DAU lift for servers running regular giveaways; 5,000+ member servers sustain weekly draws without fatigue), and the dominant GiveawayBot UX is exactly button entry — which maps 1:1 onto this bot's component-interaction model with zero privileged-intent or modal cost. Implementation is a near-clone of two modules that already exist (role-menus interaction handling + announcements scheduler job). Anti-abuse is straightforward (unique entry constraint, role gating, per-button is inherently one-per-user). Main risks are alt/farming abuse (mitigate with required-role + account-age gating) and the need for a verifiably fair RNG and audit log. Strong, immediate build.

### Suggestions / feedback board with voting

- **Description:** A structured feedback channel where members submit suggestions that the bot posts as numbered embeds with upvote/downvote buttons and a live tally. Mods can approve/deny/implement with a status change and an optional response, and a leaderboard surfaces the most-wanted ideas. Admin page lists all suggestions with filters.
- **Discord mechanics:** /suggest <text> creates a suggestion row and posts an embed to the configured board channel with three buttons: Upvote (customId 'sugg:up:<id>'), Downvote ('sugg:down:<id>'), and a mod-only 'Manage' button gated by defaultMemberPermissions / a role check on event.userRoleIds. Votes are stored one-row-per-user (unique on suggestion+user) so a click toggles/switches the vote; the handler recomputes the net score and editMessage updates the embed tally. Status changes (approved/denied/implemented) are mod subcommands (/suggest approve id reason) that recolor the embed and post the mod's response. event.reply() gives ephemeral 'Vote recorded'. THE NOTABLE LIMITATION: free-text submission ideally wants a modal, which this bot does NOT support — so submission must be the slash-command text option (slightly clunkier and the text is visible in the command). Everything else is buttons + Postgres, no privileged intent.
- **Example commands:** `/suggest text:"Add a music-request channel"`, `/suggest approve id:17 reason:"Great idea, shipping next week"`, `/suggest deny id:17 reason:"Out of scope"`, `/suggest top`
- **External deps:** None. Postgres + buttons only.
- **Privileged intent:** No. Submission via slash command, voting via buttons, mod actions via subcommands — none touch message content or member intents.
- **Permissions:** SendMessages, EmbedLinks. Mod actions gated via defaultMemberPermissions (e.g. ManageGuild) and/or userRoleIds check.
- **Overlaps existing:** No overlap. Distinct from polls (open-ended, persistent, mod-curated, status lifecycle) and from custom-commands. Reuses the role-menus vote-toggle/edit-in-place pattern.
- **Recommendation:** **later** — A proven engagement-and-retention feature — Carl-bot's suggestion system is a flagship reason developer/gaming servers add it, and giving members a visible feedback loop (vote, get a mod response, see ideas shipped) directly drives the return-visit/reply-depth retention metrics that matter in large communities. Solid fit and low risk. Held at 'later' rather than 'now' for two reasons: (1) fun-factor is moderate compared with giveaways/polls — it's a governance tool, not a hype tool, so it pays off only once a server is large enough to generate idea volume; (2) the lack of modal support means submission UX is a slash-command text option, which is a real (if minor) downgrade versus competitors' modal forms. Build after the higher-fun items, ideally bundling a future modal adapter to polish the submission flow.

### Polls (button voting with live tallies)

- **Description:** Create multi-option polls answered via buttons (or a select menu for many options) with a live-updating tally embed. Differentiators over Discord's native poll: anonymous voting, durations beyond 7 days, scheduling/recurring polls, role-restricted voting, more than 10 options, and a results export — the features native polls explicitly lack.
- **Discord mechanics:** /poll create (question, options, duration, anonymous?, allowed-role?, multi-select?) posts an embed with one button per option (customId 'poll:<id>:<optIdx>') for up to 5, or a string select menu for 6-25 options. Each vote is a ComponentInteractionEvent recorded one-row-per-user (unique constraint enforces single vote or N votes for multi-select), then editMessage refreshes the tally bars on a throttle. Anonymity is trivial because votes live in Postgres, not as visible reactions — a genuine edge over native polls. Role gating reads event.userRoleIds. A scheduler-tick job closes polls at duration end, freezes the embed, and (optionally) announces the winner; recurring polls re-post on a cron-like cadence using the scheduled-messages pattern. event.reply() confirms ephemerally. No modal, no reaction intent, no privileged intent.
- **Example commands:** `/poll create question:"Next game night?" options:"Valorant, Among Us, Minecraft" duration:48h anonymous:true`, `/poll create question:"Server name change?" options:"..." allowed_role:@Member multi:false`, `/poll close id:9`, `/poll results id:9`
- **External deps:** None. Postgres + buttons/selects + scheduler.
- **Privileged intent:** No. Voting is via components; counting needs no message content.
- **Permissions:** SendMessages, EmbedLinks only.
- **Overlaps existing:** Overlaps heavily with Discord's NATIVE /poll (shipped 2024), which covers ~80% of casual polling with zero setup. Distinct from this bot's other modules. Differentiation must come from anonymous/long/scheduled/recurring/role-gated/>10-option polls — the documented native gaps.
- **Recommendation:** **later** — Polls are consistently cited as the highest-frequency, lowest-effort engagement primitive in communities, and they fit this bot's button/select model perfectly with near-zero tech risk. BUT Discord shipping native polls in 2024 has substantially eroded the case for a basic bot poll — sources note native /poll has made generalist-bot poll commands 'somewhat obsolete' for casual use. The value now lives entirely in the features native polls lack: anonymity (easy here since votes are in Postgres, not visible reactions), durations >7 days, scheduling, recurring polls, role-restricted voting, and 11-25 options. So it's a strong build ONLY if scoped to those differentiators rather than reimplementing the native experience. 'later' (not 'now') because giveaways deliver more incremental engagement per build-hour given the native-poll substitute already exists; 'not reject' because the anonymous/recurring/scheduled niche is real and very cheap to add to this stack.

### Event / game-night scheduler with RSVP

- **Description:** Schedule events (game nights, movie nights, AMAs) that post an embed with RSVP buttons (Going / Maybe / Not going), track attendee lists live, send a reminder before start, and optionally grant an attendee role for channel access. An admin page lists upcoming events.
- **Discord mechanics:** /event create (title, time, description, optional attendee-role) parses time (luxon, already in catalog) and posts an embed with three RSVP buttons (customId 'event:rsvp:<id>:going|maybe|no'). Clicks are ComponentInteractionEvents that upsert the member's status one-row-per-user and editMessage refreshes the three attendee lists. A scheduler-tick job sends a 'starting in 15m' reminder (sendMessage pinging the Going list via allowMentions.users) and marks events complete. Optional attendee-role uses GuildService.addRole on RSVP (ManageRoles + hierarchy). KEY LIMITATIONS to flag: (1) this bot's GuildService has NO native-Discord-Scheduled-Events API — it cannot create the real Discord event object that shows in the server's Events tab, so this is an embed-based RSVP, not a native event; (2) timezone handling across a global community is non-trivial (store UTC, render with Discord <t:unix:F> timestamps to sidestep per-user TZ); (3) recurring events add cron complexity. No modal/reaction/privileged intent needed for the core flow.
- **Example commands:** `/event create title:"Among Us Night" time:"2026-07-01 20:00 UTC" attendee_role:@GameNight`, `/event rsvp (via buttons)`, `/event list`, `/event cancel id:5`
- **External deps:** None required for embed-based RSVP. (Google Calendar sync, which Apollo/Sesh offer, would be external and is out of scope.)
- **Privileged intent:** No for the embed+button RSVP flow. (Reading the native Discord Events tab would need GuildScheduledEvents handling, which is adapter work, not a privileged gateway intent.)
- **Permissions:** SendMessages, EmbedLinks; ManageRoles only if the attendee-role option is used. Creating NATIVE scheduled events would need ManageEvents + new adapter methods (not currently in GuildService).
- **Overlaps existing:** RSVP-role grouping partially overlaps role-menus (self-assign role). The scheduler/reminder mechanics overlap announcements + scheduled-messages. The event concept itself is new. Competes with Discord's native Scheduled Events plus entrenched specialists (Apollo ~300k servers, Sesh, Raid-Helper).
- **Recommendation:** **later** — Recurring events (game nights, AMAs, trivia) are repeatedly named as a top retention driver, and RSVP tracking is the feature members actually want. The button-RSVP core fits this bot well. But it scores lower on fit/now-readiness than giveaways for three concrete reasons: (1) the category is dominated by deep specialists (Apollo on 300k+ servers, Sesh, Raid-Helper) whose value is in timezone handling, recurring signups, custom roles, and Google Calendar sync — hard to match cheaply; (2) Discord's own native Scheduled Events cover the basic 'announce an event with interested-count' need for free, and this bot cannot create those native events without new adapter work; (3) correct timezone + recurrence handling is the real engineering cost and the main source of tech risk. A worthwhile 'later' build scoped to single-server embed RSVP + reminders + Discord timestamp rendering, explicitly NOT trying to out-feature Apollo.

### Random team / group generator

- **Description:** Split a set of participants into N random (optionally size-capped) teams for game nights, scrims, or activities, and post the rosters as an embed. Participants are gathered via an 'I'm in' button or passed as a mention list; supports excluding users and re-shuffling.
- **Discord mechanics:** /teams make (count or size) either takes a list of mentioned users from the command options, or posts a 30-second 'Join lobby' button (customId 'teams:join:<id>') collecting clickers as ComponentInteractionEvents into a temp roster, then shuffles (crypto RNG) and posts an embed with the team rosters. A 'Reshuffle' button reruns the split. CRITICAL CAPABILITY GAP: the popular variant of this feature MOVES players into separate team voice channels — but this bot's GuildService exposes NO voice-member-move or voice-channel-create methods, and there are NO voice-state platform events surfaced to modules. So the auto-move-to-voice-channels feature is OUT OF SCOPE without new adapter work; only a text/embed roster is feasible. The text version needs only buttons + RNG, no persistence beyond the ephemeral lobby, no privileged intent.
- **Example commands:** `/teams make players:@a @b @c @d @e @f teams:2`, `/teams lobby size:5 (posts a Join button)`, `/teams reshuffle`
- **External deps:** None.
- **Privileged intent:** No for the text-roster version. The desirable voice-move version would need NEW voice-state adapter capabilities (not a gateway intent per se, but unbuilt adapter surface).
- **Permissions:** SendMessages, EmbedLinks for text rosters. The voice-move variant would need MoveMembers + ManageChannels AND new GuildService methods that do not exist today.
- **Overlaps existing:** No overlap with existing modules. It is a natural companion to the Event/game-night scheduler. Standalone it is a thin utility.
- **Recommendation:** **later** — A genuinely fun, low-risk utility for game-night servers, but it is a niche, narrow-audience feature rather than a broad engagement driver — community-impact is limited to the subset of servers that actually organize team games, and the existing third-party options are tiny single-purpose bots, signaling modest demand. Its biggest draw in the wild (auto-distributing players into team voice channels) is blocked by this bot's lack of any voice-member-move / voice-channel adapter surface and lack of voice-state events, so a faithful implementation is text-roster-only and noticeably less magical. Best shipped as a small add-on bundled with the Event scheduler ('later'), not as a standalone priority. Not a reject — it is cheap and pleasant — but it does not earn a 'now' slot on its own merits.

### Tournament brackets

- **Description:** Run single/double-elimination (or round-robin) tournaments inside Discord: open signups, seed entrants, generate a bracket, let players/mods report match results that auto-advance winners, and render the current bracket. Aimed at competitive gaming communities.
- **Discord mechanics:** /tournament create + a Join button for signups (ComponentInteractionEvent -> entrants table), then /tournament start seeds and generates a bracket stored as match rows in Postgres. Results are reported via /match report or Win/Win buttons on each match embed (customId 'match:<id>:winner'), and the service advances the winner and posts the next matchup. Bracket VISUALIZATION is the hard part: a text/embed bracket is feasible for small fields, and this bot already has @resvg/resvg-js (used by dynamic-cards) to render an SVG bracket to PNG for larger fields — a real advantage over text-only competitors. HOWEVER the domain logic is heavy: seeding, byes, double-elimination losers-bracket routing, dispute/score handling, and many stateful edge cases. No privileged intent or modal strictly required (signups/results via buttons), though score entry would benefit from a modal this bot lacks.
- **Example commands:** `/tournament create name:"Summer Smash" format:single_elim max:16`, `/tournament start id:3`, `/match report match_id:12 winner:@playerB`, `/tournament bracket id:3`
- **External deps:** None required (self-contained brackets); could optionally integrate Challonge API, but that adds a fragile external dependency the platform guidance says to avoid.
- **Privileged intent:** No. Signups and result reporting are component-driven.
- **Permissions:** SendMessages, EmbedLinks; AttachFiles if rendering a bracket image. No role/voice perms for the core flow.
- **Overlaps existing:** No overlap with existing modules. Competes with mature specialists (Tourney Bot, Challonge bots) that already run 100+ player tournaments and support every bracket format.
- **Recommendation:** **later** — High fun-factor for the right (competitive-gaming) community and a strong retention tool there, and the bot's existing resvg image pipeline means it could render real bracket images — a differentiator. But it is by far the most complex and highest-risk candidate in the group: correct seeding/byes, double-elimination losers-bracket routing, match-dispute handling, and a large stateful surface make it a multi-week build with many edge cases, versus the days-long button modules around it. Its audience is also narrower (competitive servers only), and entrenched specialists already cover it well. Recommend 'later': only build once the cheaper, broader-impact items (giveaways, polls, suggestions) are shipped, and scope V1 tightly to single-elimination with mod-reported results before attempting double-elim or auto-seeding. Not a reject — it is genuinely engaging for gaming communities — but it is the wrong thing to build first.

### Predictions / 'Call It' event predictions (ADDED)

- **Description:** A Twitch-Predictions-style mini-game: a mod opens a prediction ('Who wins tonight's scrim?') with 2-4 outcomes; members lock in a guess via buttons before a deadline; when the mod resolves the correct outcome, everyone who called it right earns points/a streak, surfaced on a leaderboard. Pure bragging-rights economy, no real wagering.
- **Discord mechanics:** /predict open (question, options, lock-in deadline) posts an embed with one button per outcome (customId 'predict:<id>:<optIdx>'); each click records the member's pick one-row-per-user (locked once the scheduler-tick job passes the deadline and edits the embed to 'LOCKED'). /predict resolve <id> <winningOpt> marks correct guessers, awards points to a predictions_score table, and posts the winners + updated leaderboard. /predict leaderboard shows top callers. Entirely buttons + Postgres + the existing scheduler-tick pattern; event.reply() confirms picks ephemerally. No modal, no reaction intent, no privileged intent, no real currency (sidesteps gambling/abuse concerns).
- **Example commands:** `/predict open question:"Who wins the scrim?" options:"Team A, Team B" lock:30m`, `/predict (members click an outcome button)`, `/predict resolve id:8 winner:"Team A"`, `/predict leaderboard`
- **External deps:** None.
- **Privileged intent:** No. Picks via buttons, resolution via mod command, scoring in Postgres.
- **Permissions:** SendMessages, EmbedLinks. Mod-only open/resolve gated via defaultMemberPermissions.
- **Overlaps existing:** No overlap. Distinct from polls (polls measure opinion with no right answer; predictions have a resolved outcome and a persistent score/leaderboard). Reuses the giveaway/poll button+scheduler scaffolding.
- **Recommendation:** **later** — Added because research repeatedly flags 'tie engagement to recurring loops and competition' as what actually retains members, and Twitch-style channel predictions are a known high-repeatability hype mechanic that maps perfectly onto this bot's button + scheduler + Postgres stack with no privileged-intent or modal cost. The points/streak leaderboard creates a return-visit loop (predict during every event, climb the board) that plain polls lack, and using bragging-rights points instead of real currency avoids the gambling/abuse and moderation headaches of a full economy. Scored slightly below giveaways and marked 'later' only because it is net-new (no existing third-party template to copy) and pays off most once recurring events exist to predict on — natural to bundle with the Event scheduler. Very strong, cheap fun feature; build after the proven giveaway/poll items.

### Random Fun Commands (8ball / dice / roll / choose / coinflip / rps)

- **Description:** A bundle of classic stateless RNG commands: /8ball (magic-8-ball answer to a question), /roll (NdM dice, e.g. 2d6), /dice (single die), /coinflip, /choose (pick randomly from a comma list), and /rps (rock-paper-scissors vs the bot via buttons). These are the lowest-common-denominator 'fun' staples every major bot (Tatsu, MEE6, Carl-bot, Dyno) ships, used as conversation lubricant and dispute-settlers.
- **Discord mechanics:** Pure slash commands with simple options (CommandOptionDef type 'string'/'integer'). All logic is in-process RNG (Math.random / crypto), no DB, no external calls. Replies are plain text or a small embed via replyRich. /rps can be enhanced with three buttons (rock/paper/scissors) whose customId routes back via ComponentInteractionEvent and resolves with an ephemeral reply — fully inside the existing button capability, no modals, no reactions, no privileged intents. /choose splits a comma-delimited string option. A per-user/per-channel cooldown (in-memory Map or a tiny Postgres throttle table) is the only anti-spam need.
- **Example commands:** `/8ball question:Will we hit 1000 members?`, `/roll dice:2d20`, `/coinflip`, `/choose options:pizza, sushi, tacos`, `/rps (then click a button)`
- **External deps:** None. All RNG is local. No API keys, no network.
- **Privileged intent:** No. Slash commands and buttons only; no MessageContent or GuildMembers needed.
- **Permissions:** None beyond default send-message / use-application-commands.
- **Overlaps existing:** No overlap with the 11 existing modules. custom-commands can emit static text/embeds but cannot do RNG, dice math, or interactive RPS, so this is genuinely new and not duplicable with current features.
- **Recommendation:** **now** — Trivially cheap, zero external risk, perfect architectural fit (slash + optional buttons, no new intents), high testability (pure functions). Research confirms these are universal staples shipped by Tatsu/MEE6/Carl-bot and act as everyday conversation lubricant. Fun-factor is only moderate (low novelty, shallow) and spam risk is real (people spam /8ball), so it needs a per-user cooldown, but the cost-to-value ratio is the best in the group. Strong immediate win — small surface, instant utility, settles the 'does this server have fun commands' baseline expectation. Score restraint: funFactor capped at 3 because these are commodity, not standout, features.

### Meme / Image Commands (external API)

- **Description:** Commands that pull random memes/images on demand — /meme (random meme image), /cat, /dog, plus optional image-macro generation (caption an image). Modeled on the meme/animal commands in Tatsu and Dank Memer, which inject lightweight, shareable content to revive dead chat.
- **Discord mechanics:** Slash command -> ctx.defer() (network call may be slow) -> fetch from a free API (e.g. D3vd Meme_Api / meme.deadbeef.io for memes, some.random.api / thecatapi for animals) -> reply with an embed carrying imageUrl via replyRich. An 'Another' button (customId routed via ComponentInteractionEvent) lets users reroll without re-typing the command — a nice button-native UX. No modals, no reactions, no privileged intents. Needs an outbound-HTTP allowlist and timeout (the audio module already establishes request-timeout patterns). NSFW must be filtered server-side using the API's nsfw flag before posting; an admin config page can toggle NSFW-allowed per channel.
- **Example commands:** `/meme`, `/meme subreddit:wholesomememes`, `/cat`, `/dog`, `/meme (then click 'Another')`
- **External deps:** Free third-party meme/image APIs (Reddit-backed meme APIs, animal-image APIs). These are the fragile part: the once-ubiquitous meme-api.herokuapp.com died when Heroku killed free dynos, and Reddit's 2023 API clampdown (100 QPM, 1000-post cap, NSFW restrictions) means Reddit-backed memes can rate-limit or go stale. Mitigate with multiple fallback sources + caching a pool of recent results in Postgres.
- **Privileged intent:** No privileged intent. (Only outbound HTTP, which is a config/allowlist concern, not a gateway intent.)
- **Permissions:** Embed Links permission; ideally restrict NSFW-capable sources to NSFW-marked channels.
- **Overlaps existing:** No overlap. dynamic-cards renders server-side PNGs (welcome cards) but does not fetch external memes; custom-commands serves static content only. Genuinely new.
- **Recommendation:** **later** — Higher fun-factor than the RNG bundle (memes are inherently shareable, drive dead-chat revival per CommunityOne/Dank Memer research) and fits the button/embed model cleanly. But it carries real external-dependency fragility (dead/rate-limited meme APIs, Reddit clampdown) and genuine content-safety/moderation risk (a random Reddit pull can surface NSFW/offensive content into a LARGE community), which is exactly the kind of fragile external API the constraints warn against. Worth doing, but it needs a hardened multi-source + NSFW-filter + caching design and per-channel admin gating first — hence 'later', not 'now'. Spam risk is high (reroll spam) so cooldowns are mandatory.

### Server Stats & Weekly Highlights Recap (scheduler)

- **Description:** Automated community analytics: a /stats command for on-demand server/user activity (messages this week, most active channels, top chatters) plus a scheduled weekly recap auto-posted to a configured channel ('This week: 4,210 messages, top chatter @X, hottest channel #memes, N new members'). Mirrors Statbot and Discord's own Server Insights, and the 'recognize most active members' loop that Arcane/Statbot use to drive retention.
- **Discord mechanics:** Subscribe to the message.create platform event (already emitted, GuildMessages intent is ON by default) and increment per-user / per-channel / per-guild counters in Postgres+Drizzle. CRUCIALLY this needs only message metadata (author id, channelId, mentionCount, hasAttachments) — NOT message text — so it works WITHOUT the privileged MessageContent intent. The existing interval Scheduler runs a weekly job that aggregates the week's rows and posts a recap embed (and resets/rolls the window) via GuildService.sendMessage. /stats and /leaderboard render embeds; a string-select menu can switch the leaderboard view (messages vs voice vs invites). Admin config page sets the recap channel + schedule + opt-out roles. Optionally render a stats card as a PNG by reusing the dynamic-cards/resvg pipeline.
- **Example commands:** `/stats`, `/stats user:@member`, `/leaderboard (then select 'Top chatters' / 'Top channels')`, `weekly auto-posted recap embed (scheduler)`
- **External deps:** None. Entirely self-hosted on the existing message.create event + Scheduler + Postgres. No third-party API.
- **Privileged intent:** No, for the core (message-COUNT) stats — message.create fires with metadata even without MessageContent. ONLY voice-activity stats and join/leave-count stats need more: voice-minutes would require new voice-state adapter work (no voice events surfaced today), and accurate member-growth counts need the privileged GuildMembers intent. Note these as scoped add-ons, not core.
- **Permissions:** Send Messages / Embed Links in the recap channel. View Channels for any channel it counts.
- **Overlaps existing:** No existing module tracks activity or posts recaps. scheduled-messages/announcements post FIXED content on a cron but compute nothing; this is the missing analytics+aggregation layer. No overlap.
- **Recommendation:** **now** — Highest community-impact in the group and an exceptional architectural fit: it uses the scheduler, Postgres, and the already-on message.create metadata stream WITHOUT any privileged intent for its core feature. Research is strong and consistent — recognizing top contributors and weekly recaps (Statbot, Arcane, Discord Insights) are proven retention drivers in large servers, and a recurring auto-post creates a reliable weekly engagement beat (repeatability 5). Low spam risk (read-only/admin-gated), high moderability. Complexity is moderate (schema design, aggregation, window rollover, opt-out for privacy) but contained. The voice-minutes and member-growth dimensions are gated behind extra adapter/intent work, so ship the message-count core now and treat those as later extensions. Top pick of the group alongside the RNG bundle.

### Soundboard (extends audio module)

- **Description:** A library of short, one-tap sound clips (airhorn, drumroll, rimshot, custom server memes) played instantly into the user's current voice channel, presented as a button grid. Mirrors Blerp and Discord's native Soundboard, which research shows is on ~2.5M servers — a heavily-used voice-channel engagement primitive.
- **Discord mechanics:** Reuses the voice stack: VoiceCapability.getUserVoiceChannel() -> join() -> VoiceSession.play(AudioStreamSource). A soundboard clip is just an AudioStreamSource whose createStream() opens a short local OGG/Opus file (inputType 'ogg-opus' avoids transcoding) — the DirectHttp provider / a new file provider supplies it; no yt-dlp needed for built-in sounds. /soundboard posts a message of buttons (one per sound, customId 'sb:<id>'); a click routes via ComponentInteractionEvent, joins the clicker's voice channel, and plays. Clips are managed via a Postgres table + admin upload page. KEY DESIGN FRICTION: the current PlayerManager is a single queue-based session per guild built for music; firing a one-shot SFX needs new engine work to either (a) interrupt/duck the music session or (b) run a separate short-lived 'one-shot' play path, and to handle concurrency when music is already playing. Buttons-only, no modals/reactions; uploads go through the existing admin panel, not a Discord modal.
- **Example commands:** `/soundboard (posts the button grid)`, `click an 'Airhorn' button`, `/sound play name:drumroll`, `/sound add (admin, via web panel upload)`
- **External deps:** None for built-in/uploaded clips (local files streamed through existing voice). Only optional if you let users pull arbitrary URLs, which adds the audio module's existing fetch/allowlist concerns.
- **Privileged intent:** No gateway privileged intent. But it DOES depend on voice, and there are currently no voice-STATE events surfaced — that's fine for play-on-click (driven by the interaction), but auto-leave-when-empty or 'is the user still in VC' edge cases may need extra voice handling.
- **Permissions:** Connect + Speak in voice channels (the audio module already requires these). Storage for uploaded clips.
- **Overlaps existing:** Extends the existing audio module rather than duplicating it — no music feature does instant SFX. Distinct from queue-based playback, but shares the voice session, which is the integration risk (single-session-per-guild contention with music).
- **Recommendation:** **later** — Genuinely fun and well-validated by research (native Soundboard on ~2.5M servers; Blerp's whole product), and it leverages real existing infrastructure (voice stack, admin uploads, Postgres). But it is the riskiest fit in the group on two axes: (1) it collides with the single-queue-per-guild audio engine — instant one-shots while music plays need real concurrency/ducking work, not a thin wrapper; and (2) spam/abuse risk is the highest in the group (airhorn spam in voice is acutely disruptive and hard to moderate — earrape, mic-blasting), demanding strict per-user voice cooldowns and a kill switch. Discord's own native Soundboard also partially cannibalizes the need. Strong 'later': do it after the cheap wins, with deliberate engine and anti-abuse design. Not a reject because it builds on assets the bot uniquely already has.

### Highlight / Clip Submission Board (button-based starboard)

- **Description:** A community 'best-of' board: members nominate a funny/great message or clip, the community upvotes it, and high-vote items get reposted to a dedicated #highlights channel. This is the classic starboard pattern (Carl-bot, Dyno, Starboard bot) that research consistently lists as a top engagement/recognition feature — but reimagined for this bot's button-only interactivity.
- **Discord mechanics:** MAJOR MECHANIC MISMATCH: the canonical starboard works by listening for a star EMOJI REACTION crossing a threshold — but this bot has NO GuildMessageReactions intent and NO reaction listener, so the standard design is OFF THE TABLE without new adapter/intent work. A button-native redesign is required: a /highlight nominate command (or, ideally, a Discord message-context-menu 'Apps' command — which is NOT modeled in the current CommandDefinition contract, so it'd be new adapter work too) captures the target message link; the bot posts a candidate embed in a review channel with an 'Upvote' button (customId routed via ComponentInteractionEvent); votes are deduped per-user in Postgres; when votes cross a configurable threshold the Scheduler or the vote handler reposts it to #highlights. Vote dedupe, link/permalink capture, and preventing self-vote/farming are the fiddly parts. Buttons + embeds + Postgres only; threshold sweeps can use the existing scheduler.
- **Example commands:** `/highlight nominate message_link:https://discord.com/channels/...`, `click 'Upvote' on a candidate`, `auto-repost to #highlights at N votes`, `/highlights top (weekly best-of, ties into the recap)`
- **External deps:** None external. Self-contained on buttons + Postgres + scheduler.
- **Privileged intent:** No privileged GATEWAY intent strictly required for the button-based nominate flow. BUT it needs capabilities the platform does not currently expose: emoji-reaction listening (GuildMessageReactions intent + a reaction adapter) for the natural UX, and/or a message-context-menu command type — both are NEW adapter/contract work flagged by the constraints as extra cost. Fetching/rendering the nominated message's content for the embed may also touch MessageContent if you want to show the original text.
- **Permissions:** Send Messages / Embed Links in the highlights + review channels; Read Message History to resolve the nominated message.
- **Overlaps existing:** No existing module does highlights/voting. Conceptually adjacent to role-menus (button/select voting plumbing can be reused) and to the weekly recap (could surface 'top highlight of the week'), but not a duplicate.
- **Recommendation:** **later** — The engagement payoff is real — starboards are a proven recognition/best-of loop (Carl-bot/Dyno/Starboard) and would pair beautifully with the weekly recap. But it has the WORST architectural fit in the group: the standard emoji-reaction mechanic is impossible without new intent+adapter work, and the most ergonomic alternative (right-click 'Apps' context-menu nominate) is a command type the current CommandDefinition contract doesn't model. The button-redesign is viable but is the most complex build here (vote dedupe, anti-farming/self-vote, link resolution, threshold sweeps) and a clunkier UX than a one-tap star. Plus abuse risk is high in a LARGE server (brigading, vote farming for clout). Not a reject — it's desirable and self-contained on persistence — but it's clearly 'later' behind the cheaper, better-fitting wins, and should be scoped only after deciding whether to invest in a reaction adapter. fitsExistingBot scored low (2) specifically because the natural mechanic needs platform extensions.

### Button Polls / Quick Votes (ADDED)

- **Description:** ADDED CANDIDATE. /poll lets any member spin up a question with 2-10 options; members vote by clicking buttons; the embed live-updates the tally and bar chart; results can auto-close on a timer. Polls are a top conversation-driver and decision tool in large servers, and Discord's own native polls validated the demand. This is the single best architectural fit not on the original list.
- **Discord mechanics:** A near-perfect match for the platform: /poll question:... options:a|b|c posts an embed with one button per option (customId 'poll:<id>:<optIdx>'); ComponentInteractionEvent handles each vote, dedupes/toggles per-user in Postgres, and the handler's update() refreshes the same message in place with updated counts and a text bar chart. The existing Scheduler closes polls at a deadline and posts final results. No modals (options come from a delimited slash option), no reactions, no privileged intents. Anonymous vs named voting is a config toggle; per-user single-vote enforcement is a unique DB constraint.
- **Example commands:** `/poll question:Game night tonight? options:Yes|No|Maybe duration:2h`, `click an option button to vote`, `/poll results id:123`, `auto-closes and posts winner via scheduler`
- **External deps:** None. Buttons + Postgres + scheduler only.
- **Privileged intent:** No. Slash + buttons + scheduler; nothing privileged.
- **Permissions:** Send Messages / Embed Links.
- **Overlaps existing:** No existing module does polls. Reuses the same button-interaction + update() plumbing as role-menus and the now-playing panel, so it slots in cleanly without new platform capabilities.
- **Recommendation:** **now** — Surfaced by research as a core interactive-engagement feature, and it is arguably the cleanest fit for THIS bot's exact strengths: live-updating button messages via update(), Postgres vote storage, scheduler-driven auto-close — zero new intents, no modals, no reactions, no fragile external API. Low spam/abuse risk (one vote per user, admin-closable), high repeatability (every event/decision becomes a poll). It outscores three of the four original candidates on fit-vs-cost. Recommending 'now' as a high-value, low-risk addition that the original group overlooked.

### Trivia / Quiz Game (ADDED)

- **Description:** ADDED CANDIDATE. /trivia posts a multiple-choice question; players race to click the correct answer button; points accrue to a Postgres leaderboard; a scheduled 'daily trivia' drops a question each day with a winner role. Modeled on TriviaBot (one of the most popular Discord game bots, 160k+ questions, OpenTDB-backed), which research highlights for driving competitive, repeatable engagement.
- **Discord mechanics:** Slash command -> fetch a question from OpenTDB (opentdb.com — a free, stable, no-key trivia API; results can be cached/preloaded into Postgres to remove runtime dependency) -> post embed + 4 answer buttons (customId 'tq:<id>:<choice>'). First correct ComponentInteractionEvent wins; the handler scores in Postgres and update()s the message to reveal the answer. The Scheduler runs daily/weekly trivia drops and can hand a 'Trivia Champ' role via GuildService.addRole, tying into autoroles. Leaderboard via embed + string-select to switch daily/weekly/all-time. No modals, no reactions, no privileged intents.
- **Example commands:** `/trivia category:Video Games`, `click the answer button`, `/trivia leaderboard (select daily/weekly/all-time)`, `daily auto-posted trivia (scheduler) + winner role`
- **External deps:** OpenTDB (free, key-less, well-established) for the question bank — and it can be mitigated to near-zero by bulk-importing questions into Postgres once, so runtime has NO hard external dependency. Far less fragile than the meme APIs.
- **Privileged intent:** No. Slash + buttons + scheduler; awarding roles uses the existing GuildService (needs Manage Roles permission, not an intent).
- **Permissions:** Send Messages / Embed Links; Manage Roles only if you award a winner role.
- **Overlaps existing:** No existing game module. Complements the weekly-recap leaderboard and can reuse autorole plumbing for the champ role. No duplication.
- **Recommendation:** **later** — Highest fun-factor in the whole group (genuine competitive game loop, daily-return hook, leaderboards — exactly the repeatable engagement TriviaBot is renowned for) and a good fit using buttons + scheduler + Postgres with only a soft, easily-cached external dependency. Slightly more build than polls (question import/caching, scoring, race-condition handling on first-correct-click, leaderboard windows), so it lands at 'later' rather than 'now' — but it is a clear high-value addition the original list missed and should rank above memes, soundboard, and highlights in the build queue.
